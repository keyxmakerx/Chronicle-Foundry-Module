# Chronicle Sync ‚Äî Foundry VTT Module

## Purpose

The Chronicle Sync module for Foundry VTT enables bidirectional real-time sync
between a Chronicle instance and a Foundry VTT game world. It runs as a Foundry
v12/v13 module, connecting via REST API + WebSocket to the Chronicle backend.

## Architecture

```
module.mjs (entry point)
  ‚îî‚îÄ SyncManager (sync-manager.mjs)
       ‚îú‚îÄ ChronicleAPI (api-client.mjs)    ‚Üê REST + WebSocket client
       ‚îú‚îÄ JournalSync (journal-sync.mjs)   ‚Üê Entity ‚Üî JournalEntry
       ‚îú‚îÄ MapSync (map-sync.mjs)           ‚Üê Map drawings/tokens ‚Üî Scene
       ‚îú‚îÄ CalendarSync (calendar-sync.mjs) ‚Üê Calendar events/date ‚Üî Calendaria/SimpleCalendar
       ‚îú‚îÄ ShopWidget (shop-widget.mjs)     ‚Üê Shop entity inventory in Foundry UI
       ‚îú‚îÄ NoteSync (note-sync.mjs)        ‚Üê Chronicle Notes ‚Üî JournalEntry
       ‚îú‚îÄ ItemSync (item-sync.mjs)        ‚Üê Actor inventory ‚Üî Chronicle relations
       ‚îú‚îÄ SyncDashboard (sync-dashboard.mjs) ‚Üê 6-tab ApplicationV2 status/management UI
       ‚îî‚îÄ ActorSync (actor-sync.mjs)       ‚Üê Character entity ‚Üî Actor
            ‚îú‚îÄ dnd5e-adapter.mjs            ‚Üê D&D 5e field mapping
            ‚îú‚îÄ pf2e-adapter.mjs             ‚Üê Pathfinder 2e field mapping
            ‚îî‚îÄ generic-adapter.mjs          ‚Üê Data-driven adapter (any system)
```

### Data Flow

1. **GM starts Foundry** ‚Üí `module.mjs` initializes SyncManager
2. **SyncManager.start()** ‚Üí validates config, initializes modules, connects WS
3. **WebSocket connects** ‚Üí performs initial sync (pulls mappings since `lastSyncTime`)
4. **Live updates** ‚Üí WS messages routed to modules; REST for CRUD operations
5. **Bidirectional** ‚Üí Foundry hooks detect local changes, push to Chronicle via REST

### Sync Guard

All sync modules use a `_syncing` boolean flag to prevent infinite loops:
- Set `true` before applying a remote change to Foundry
- Foundry's change hooks fire but see `_syncing === true` ‚Üí skip pushing back
- Reset to `false` after the change is applied

## WebSocket Message Types

See API-CONTRACT.md for the full WebSocket protocol. Summary of handling status:

**Handled:**
- `entity.created/updated/deleted` ‚Üí JournalSync
- `entity_type.created/updated/deleted` ‚Üí JournalSync (folder updates)
- `drawing.created/updated/deleted` ‚Üí MapSync
- `token.created/moved/updated/deleted` ‚Üí MapSync
- `marker.created/updated/deleted` ‚Üí MapSync
- `fog.updated` ‚Üí MapSync (renders fog regions as overlay drawings)
- `calendar.event.created/updated/deleted` ‚Üí CalendarSync
- `calendar.date.advanced` ‚Üí CalendarSync
- `note.created/updated/deleted` ‚Üí NoteSync
- `relation.created/deleted/metadata_updated` ‚Üí ItemSync
- `sync.status` ‚Üí API client (connection state)

**Not yet handled (available per contract):**
- `calendar.season.changed` ‚Äî Season boundary crossed
- `calendar.moon.phase_changed` ‚Äî Moon phase changed
- `calendar.weather.changed` ‚Äî Weather set or generated
- `calendar.structure.updated` ‚Äî Calendar structure modified (months, weekdays, etc.)
- `calendar.era.changed` ‚Äî Era boundary crossed
- `layer.updated` ‚Äî Map layer changed
- `sync.error` ‚Äî Synchronization error
- `sync.conflict` ‚Äî Data conflict detected

## Security

- API key stored as Foundry world setting (GM-only visible, masked as password input)
- REST uses `Authorization: Bearer <key>` header
- WebSocket uses `?token=<key>` query param (necessary for WS upgrade)
- Per-key rate limiting enforced server-side (60 req/min default)
- Campaign-scoped: key can only access its bound campaign
- Device fingerprint binding: optional one-time device lock

## Known API Field Discrepancies

The API contract (API-CONTRACT.md) defines the canonical field names. The code
currently uses some legacy/alternate names that the API also accepts. These
should be verified against the live API and aligned if possible:

| Area | Contract Field | Code Uses | Files |
|------|---------------|-----------|-------|
| Entity content (GET) | `content` | `entry_html` | `journal-sync.mjs` |
| Entity content (POST) | `content` | `entry` | `journal-sync.mjs` |
| Entity visibility (GET) | `visibility: "public"` | `is_private: true/false` + `visibility` | `journal-sync.mjs`, `actor-sync.mjs` |
| Sync mapping POST | `type`, `foundry_id` | `chronicle_type`, `external_id`, `external_system`, `sync_direction` | All sync modules |
| Sync lookup query | `?foundry_id=X&type=Y` | `?external_system=foundry&external_id=X` | `sync-manager.mjs` |
| Relations (WS payload) | `source_id`, `target_id`, `relation_type_id` | `sourceEntityId`, `targetEntityId`, `relationType` | `item-sync.mjs` |
| Calendar events (POST) | `color`, `icon`, `all_day`, `recurrence_*`, `start_hour`, `start_minute` | Not sent (uses simplified fields) | `calendar-sync.mjs` |

**Note:** These may represent legitimate API aliases (the API accepts both forms)
or WS vs REST payload differences. Verify with the Chronicle dev before changing.

## Known Limitations

- **Fog of war**: One-way sync (Chronicle ‚Üí Foundry). Fog regions rendered as
  semi-transparent polygon drawings on the Foundry scene. No Foundry ‚Üí Chronicle
  push yet (Foundry's native fog is vision-based, Chronicle's is polygon-based)
- **SimpleCalendar events**: Limited ‚Äî events managed as journal notes, no CRUD hooks
- **Shop icon field**: Always returns null (`shop-widget.mjs:146`)
- **Single scene**: Only the active Foundry scene syncs (no multi-scene)
- **GM only**: Full sync runs only for GM users; players get passive updates via Foundry
- **Character sync: limited fields back from PF2e**: Only HP and name sync from Chronicle to PF2e actors (most values are derived from items/rules)
- **Character sync: custom system requirements**: Custom game systems must include `foundry_path` annotations on their character preset fields for bidirectional sync. Systems without annotations get no character sync (the generic adapter returns null). dnd5e and pf2e have hand-written adapters that handle edge cases.
- **System matching fallback**: `SYSTEM_MAP_FALLBACK` (3 entries: dnd5e, pf2e, drawsteel) in sync-manager.mjs is used only when the `/systems` API call fails or the system lacks a `foundry_system_id` in its manifest. Normal operation queries the API and matches dynamically.

## Features

### F-1: Journal Sync Fidelity (DONE)
- Split entity `entry_html` by `<h1>`/`<h2>` headings into multiple Foundry pages
- Concatenate all text pages back into single `entry` for Foundry‚ÜíChronicle
- Track page mapping via `chronicle-sync.pageMap` flag
- Detect ownership changes in `updateJournalEntry` hook, push to Chronicle

### F-2: Granular Permission Mapping (DONE)
- Map Chronicle `visibility: 'custom'` + `entity_permissions` to per-user Foundry ownership
  - Chronicle `view` ‚Üí Foundry `OBSERVER`, `edit` ‚Üí `OWNER`
- Uses `GET/PUT /entities/:eid/permissions` endpoints
- Reverse-map Foundry ownership changes to Chronicle entity permissions

### F-3: System Detection & Character Field Templates (DONE)
- `SYSTEM_MAP` in sync-manager.mjs: `{ dnd5e: "dnd5e", pf2e: "pathfinder2e", drawsteel: "drawsteel" }`
- `SyncManager._detectSystem()` queries `GET /api/v1/campaigns/:id/systems` on start
- Matches Foundry `game.system.id` to Chronicle system, stores in `detectedSystem` setting
- `syncCharacters` boolean setting (gated on system match)
- Dashboard Status tab shows system match info and character sync availability

### F-4: Actor ‚Üî Entity Sync (DONE)
- `actor-sync.mjs`: Bidirectional sync between Foundry Actors and Chronicle character entities
- System adapters: `adapters/dnd5e-adapter.mjs` (15 fields), `adapters/pf2e-adapter.mjs` (HP/name back only)
- Hooks: `createActor`, `updateActor`, `deleteActor` with `_syncing` guard
- WS: `entity.created/updated/deleted` filtered by character type (via type_slug, type_name, or type_id)
- Dashboard "Characters" tab: synced/unlinked actors, Push button, empty states
- Delete from Chronicle unlinks Actor (preserves data); delete from Foundry deletes Chronicle entity
- dnd5e/pf2e adapters remain as overrides; generic adapter handles all other systems (see F-4.5).

### F-QoL: Foundry Sync Diagnostics & Error Handling (DONE)
- API client health metrics: success/error counts, uptime percentage, reconnect attempts, last timestamps
- Structured error log (last 50 entries with timestamp, method, path, status, message)
- Retry queue for failed write operations (processes on reconnect, max 3 retries, cap 50 items)
- Dashboard Status tab: diagnostics grid, error log, field mapping debug info, activity log

### F-4.5: Generic System Adapter & Dynamic System Matching (DONE)
- `generic-adapter.mjs`: Fetches field definitions from `/systems/:id/character-fields` API. Auto-generates `toChronicleFields()`/`fromChronicleFields()` mappings from `foundry_path` annotations. Respects `foundry_writable` flag. Type casting for numbers.
- `_detectSystem()` in sync-manager.mjs now API-driven: queries `/systems`, matches by `foundry_system_id`. Falls back to `SYSTEM_MAP_FALLBACK` for legacy support.
- `_loadAdapter()` in actor-sync.mjs tries built-in adapters (dnd5e, pf2e) first, then falls back to generic adapter via `createGenericAdapter()`.
- Custom system workflow: Chronicle systems with `foundry_system_id` and `foundry_path` annotations ‚Üí module auto-detects, reads field defs from API, syncs characters without adapter code.
- **Limitation**: Systems without `foundry_path` on their fields get no character sync.

### F-5: NPC Visibility Sync (DONE)
- NPC visibility changes sync bidirectionally ‚Äî Chronicle `is_private` ‚Üî Foundry `prototypeToken.hidden`

### F-6: Armory / Inventory System (DONE)
- `item-sync.mjs`: Actor inventory ‚Üî Chronicle "Has Item" relations
- Relation metadata stores quantity, equipped status
- System item field definitions loaded from `/systems/:id/item-fields` API
- Hooks: `createItem`, `updateItem`, `deleteItem` with `_syncing` guard

### F-7: Shop / Marketplace Enhancement (DONE)
- `shop-widget.mjs`: Shop window UI (ApplicationV2) with drag-to-character-sheet
- Purchase flow: creates "Has Item" relation + inventory item on actor
- Real-time inventory updates via WS events
- Context menu on shop-type JournalEntries to open shop window

### F-8: Notes Sync (DONE)
- `note-sync.mjs`: Chronicle Notes ‚Üî JournalEntries in "Chronicle Notes" folder tree
- Separate resource from entities (different API, different folder)
- Case normalization: Chronicle Notes API uses camelCase, requests use snake_case
- Sharing ‚Üí Foundry ownership mapping (is_shared ‚Üí OBSERVER, private ‚Üí NONE)
- Folder hierarchy preserved (note folders ‚Üí Foundry subfolders)

### API Client: ConflictError (DONE)
- `api-client.mjs` exports `ConflictError` class for 409 responses
- Thrown when `expected_updated_at` is stale (optimistic concurrency)
- JournalSync and ActorSync handle conflicts per `conflictResolution` setting
- Strategies: `chronicle` (re-pull), `foundry` (force push), `newest` (compare timestamps)
