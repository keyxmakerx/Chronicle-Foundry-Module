# Timeline Plugin

## Purpose

Interactive visual timelines for campaigns. Each campaign can have multiple
timelines that display events in a zoomable, draggable D3.js visualization.
Timelines support two types of events:
- **Linked calendar events:** Many-to-many links to calendar events (original design).
- **Standalone events:** Events created directly on the timeline, no calendar required.
Timelines can optionally reference a calendar (for linking), or be calendar-free
(standalone events only). Entity groups organize events into swim-lanes.

## Tier

Plugin (Feature app with handler/service/repo/templates)

## Domain Concepts

- **Timeline:** A named visual timeline within a campaign. Optionally references
  a calendar for linking calendar events. Has its own color, icon, visibility, zoom config.
- **Event Link:** A many-to-many join between timelines and calendar events. Supports
  per-link overrides for label, color, and visibility. Source="calendar".
- **Standalone Event:** An event owned directly by a timeline (1:N). Has full parity
  with calendar events (multi-day, times, recurrence). Source="standalone".
- **Entity Group:** A named group of entities used for swim-lane organization. Events
  linked to entities in a group appear in that group's lane.
- **Visibility Rules:** JSON-based per-user visibility overrides (allowed_users,
  denied_users) for fine-grained access control beyond everyone/dm_only.
- **Zoom Levels:** era, century, decade, year, month, day — each with distinct
  visual styles in the D3.js visualization (Sprint 3+).

## Design Decisions

- **ADR-018:** D3.js chosen over Leaflet.js for timeline visualization (Leaflet is
  geographic-only; D3 provides time scales, SVG rendering, zoom/pan, tree layouts)
- **Lens over calendar events:** Timeline events are links (not copies) to calendar
  events, preserving calendar as single source of truth
- **Optional calendar:** Each timeline has a nullable calendar_id FK. When set,
  users can link calendar events. When null, only standalone events are available.
- **Standalone events are 1:N:** Standalone events belong to exactly one timeline
  (not shareable across timelines). Calendar event linking remains M:N.
- **Unified event view:** Service merges linked calendar events and standalone events
  into a single []EventLink slice (via ToEventLink), sorted by date. The Source field
  ("calendar" or "standalone") lets the frontend dispatch actions to correct endpoints.
- **Visibility rules as JSON:** Extends existing everyone/dm_only system with
  per-user allowed/denied lists. Applied to timelines, event links, and standalone
  events. Calendar events also gained per-user rules (migration 000037).

## Security & Validation

- **IDOR protection**: `requireTimelineInCampaign()` verifies timeline belongs to
  campaign. Standalone event handlers verify event's `timeline_id` matches. Entity
  group operations are scoped to `timeline_id` in SQL queries to prevent cross-timeline
  manipulation.
- **HTML sanitization**: Standalone event `description_html` is sanitized via
  `sanitize.HTML()` on both Create and Update paths (same pattern as calendar events).
- **Input validation**: Timeline names (max 255), group names (max 200), event names
  (max 255), icon pattern (`^fa-[a-z0-9-]+$`), hex color pattern, visibility enum
  (everyone/dm_only), visibility_rules JSON validated on all write paths.
- **Visibility filtering**: `canUserView()` applies role-based + per-user JSON rules.
  Malformed JSON fails open with `slog.Warn` (validated on write path). Owners always
  see everything. View-as-player mode supported via `effectiveRole()`.
- **Role enforcement**: Route middleware + CSRF tokens on all state-changing endpoints.

## Known Limitations

- **Time fields not in unified view:** `ToEventLink()` maps name, date, category,
  visibility, and entity fields but does not map time/duration fields (start_hour,
  end_year, etc.) since `EventLink` was designed for calendar event links which
  lack these. The D3 viz and event list don't display time info for any events.
- **Recurring events**: Fields stored but no server-side expansion logic.
- **No event duplication**: Events must be manually recreated.
- **LinkAllEvents race condition**: If two users bulk-link simultaneously, duplicate
  key errors may cause partial linking. Acceptable for the current usage pattern.
