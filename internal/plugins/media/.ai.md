# Media Plugin

## Purpose

The media plugin handles file uploads, storage, serving, and thumbnail generation
for Chronicle. It supports entity images, avatars, backdrops, and general
attachments. Files are stored on the local filesystem with UUID-based naming to
prevent path traversal. Integrates with the settings plugin for dynamic storage
quotas per campaign and per user.

## Architecture

```
Handler (handler.go)
  ├── Upload()         POST /media/upload
  ├── Serve()          GET  /media/:id
  ├── ServeThumbnail() GET  /media/:id/thumb/:size
  ├── Info()           GET  /media/:fileID/info
  └── Delete()         DELETE /media/:fileID
         │
Service (service.go)
  ├── Upload() — MIME validation, magic byte check, quota enforcement,
  │              UUID filename, thumbnail generation, DB insert
  ├── checkQuotas() — per-file, per-campaign, per-user storage limits
  └── generateThumbnail() — Catmull-Rom interpolation, decompression bomb check
         │
Repository (repository.go)
  ├── Create/FindByID/Delete — single file CRUD
  ├── ListByCampaign — paginated campaign-scoped list
  ├── ListAll — all files with uploader names (admin)
  ├── GetStorageStats — aggregate stats by usage_type
  └── GetCampaignUsage — total bytes + file count for quotas
```

### Dependencies

- **Uses:** settings plugin (`StorageLimiter` interface for dynamic quotas)
- **Used by:** entities plugin (entity images), maps plugin (map images),
  admin plugin (`/admin/storage` page), campaigns plugin (campaign images)

## Data Model

### `media_files` Table

| Column | Type | Description |
|--------|------|-------------|
| `id` | CHAR(36) PK | UUID |
| `campaign_id` | CHAR(36) NULL | FK to campaigns (NULL for avatars/backdrops) |
| `uploaded_by` | CHAR(36) NOT NULL | FK to users |
| `filename` | VARCHAR(500) | Date-based path: `2006/01/{uuid}.ext` |
| `original_name` | VARCHAR(255) | User's original filename |
| `mime_type` | VARCHAR(100) | Validated MIME type |
| `file_size` | BIGINT | Bytes |
| `usage_type` | VARCHAR(50) | `entity_image`, `attachment`, `avatar`, `backdrop` |
| `thumbnail_paths` | JSON | Map: `{"300": "path", "800": "path"}` |
| `created_at` | TIMESTAMP | Auto |

### Entity Integration

Entities reference media via `image_path` column (stores media file UUID).
No reverse-reference tracking exists — media files don't know which entities
use them. This is a known gap (see Known Limitations).

## API Endpoints

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| POST | `/media/upload` | Auth + Rate limit (30/min) | Upload file, returns `{id, url, thumbnail_url}` |
| GET | `/media/:id` | Public (UUID-guessable) | Serve file with 1-year cache headers |
| GET | `/media/:id/thumb/:size` | Public | Serve thumbnail (sizes: 300, 800 only) |
| GET | `/media/:fileID/info` | Auth + owner/admin | File metadata JSON |
| DELETE | `/media/:fileID` | Auth + owner/admin | Delete file and DB record |

**Entity image update:** `PUT /campaigns/:id/entities/:eid/image` (in entities plugin)
accepts `{image_path: media_uuid}` and updates the entity's image reference.

## Upload Flow

1. Client sends multipart form with `file` and `usage_type` fields
2. Handler extracts file, CSRF token validation via middleware
3. Service validates: MIME type allowlist (JPEG, PNG, WebP, GIF)
4. Service validates: magic bytes match claimed MIME type
5. Service checks: static file size limit + dynamic quotas via `StorageLimiter`
6. Service generates UUID filename in date-based directory: `{mediaPath}/2006/01/{uuid}.ext`
7. Service writes file to disk
8. Service generates thumbnails (300px, 800px) using Catmull-Rom interpolation
9. Service inserts DB record with thumbnail paths as JSON
10. Handler returns 201 with `{id, url, thumbnail_url, mime_type, file_size}`

## Security Considerations

- **MIME validation**: Allowlist only (JPEG, PNG, WebP, GIF)
- **Magic byte validation**: Prevents Content-Type spoofing
- **Decompression bomb protection**: Max 10,000x10,000px images
- **Rate limiting**: 30 uploads/min per IP
- **Storage quotas**: Per-file, per-campaign, per-user limits via settings plugin
- **UUID filenames**: Prevents path traversal, provides unguessable URLs
- **Ownership checks**: Info/Delete require uploader or site admin
- **Public serving**: Files served without auth — relies on UUID entropy (122 bits)

## Known Limitations & Future Work

- **No "referenced by" tracking**: Media files don't know which entities use them.
  Need `FindReferencingEntities()` repo method that scans `entities.image_path`.
- **No campaign owner media browser**: Owners can upload but can't browse/manage
  their files. Need `/campaigns/:id/media` page.
- **No ClamAV integration**: Files are not scanned for malware.
- **No SVG support**: SVGs blocked due to XSS risk.
- **Full page reload after upload**: `image_upload.js` does `window.location.reload()`.
  Could be improved with HTMX swap.
- **No tests**: Zero unit tests for media service.
