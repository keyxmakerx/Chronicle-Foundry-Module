# WebSocket Package (`internal/websocket/`)

## Purpose

Campaign-scoped WebSocket hub for real-time bidirectional communication between
Chronicle and external clients (Foundry VTT module, browser sessions). Broadcasts
domain events (entity changes, map updates, calendar advances) to all connections
in a campaign without polling.

## Architecture

```
HandleUpgrade (handler.go)
  → MultiAuthenticator (auth.go)       ← API key or session cookie
  → Hub.RegisterClient (hub.go)        ← creates Client, starts goroutines
       ├─ Client.readPump (client.go)  ← reads WS frames, enforces scope
       ├─ Client.writePump (client.go) ← writes queued messages, sends pings
       └─ Hub.Run() event loop         ← register/unregister/broadcast
```

### Message Flow

1. **Inbound (client → server)**: readPump reads JSON → enforces campaign scope → hub broadcast
2. **Outbound (server → client)**: service calls EventBus.Publish → hub encodes → writePump sends
3. **Echo suppression**: hub skips sender (msg.SenderID == client.ID) to prevent loops

## Files

| File | Purpose |
|------|---------|
| `message.go` | Package doc, MessageType constants (18 types), Message struct, encode/decode |
| `hub.go` | Hub manages all clients by campaign. Sequential event loop avoids races |
| `client.go` | Client struct, readPump/writePump goroutines, ping/pong keep-alive |
| `handler.go` | HTTP→WS upgrade handler, Authenticator interface, origin validation |
| `auth.go` | MultiAuthenticator: API key (query param) then session cookie fallback |
| `eventbus.go` | EventBus interface for services, hubEventBus wrapper, NoopEventBus for tests |

## Connection Parameters

| Parameter | Value | Notes |
|-----------|-------|-------|
| writeWait | 10s | Time to write a message |
| pongWait | 60s | Time to receive pong |
| pingPeriod | 30s | Interval for server-sent pings |
| maxMessageSize | 64KB | Rejects larger frames |
| sendBufferSize | 256 | Per-client outgoing channel buffer |
| broadcastBuffer | 256 | Hub broadcast channel buffer |

## Authentication

Two strategies via `MultiAuthenticator`:

1. **API Key** (Foundry VTT): `?token=<key>` query param → syncapi service validates
   - Returns: campaignID, userID (key owner), role=3 (owner), source="foundry"
2. **Session Cookie** (Browser): session cookie + `?campaign=<uuid>` query param
   - Returns: campaignID, userID, role (from campaign membership), source="browser"
3. **Bearer Fallback**: `Authorization: Bearer <key>` header if no campaign param

## Origin Validation

The WebSocket upgrader validates the `Origin` header against the configured
`BaseURL` and any additional allowed origins passed at startup. This prevents
cross-site WebSocket hijacking while allowing Foundry VTT connections from
different hosts when they authenticate with a valid API key.

## Security Considerations

- Campaign scope enforced in readPump (msg.CampaignID = client.CampaignID)
- Sender ID set server-side (msg.SenderID = client.ID) — cannot be spoofed
- Backpressure: clients with full send buffers are disconnected
- Message type validation rejects unknown/malformed types
- Origin validation prevents cross-site WebSocket hijacking
- All auth checked before WS upgrade (no unauthenticated connections)
