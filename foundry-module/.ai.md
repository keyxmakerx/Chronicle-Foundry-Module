# Foundry VTT Module (`foundry-module/`)

## Purpose

The Chronicle Sync module for Foundry VTT enables bidirectional real-time sync
between a Chronicle instance and a Foundry VTT game world. It runs as a Foundry
v12/v13 module, connecting via REST API + WebSocket to the Chronicle backend.

## Architecture

```
module.mjs (entry point)
  ‚îî‚îÄ SyncManager (sync-manager.mjs)
       ‚îú‚îÄ ChronicleAPI (api-client.mjs)    ‚Üê REST + WebSocket client
       ‚îú‚îÄ JournalSync (journal-sync.mjs)   ‚Üê Entity ‚Üî JournalEntry
       ‚îú‚îÄ MapSync (map-sync.mjs)           ‚Üê Map drawings/tokens ‚Üî Scene
       ‚îú‚îÄ CalendarSync (calendar-sync.mjs) ‚Üê Calendar events/date ‚Üî Calendaria/SimpleCalendar
       ‚îú‚îÄ ShopWidget (shop-widget.mjs)     ‚Üê Shop entity inventory in Foundry UI
       ‚îú‚îÄ SyncDashboard (sync-dashboard.mjs) ‚Üê 6-tab ApplicationV2 status/management UI
       ‚îî‚îÄ ActorSync (actor-sync.mjs)       ‚Üê Character entity ‚Üî Actor
            ‚îú‚îÄ dnd5e-adapter.mjs            ‚Üê D&D 5e field mapping
            ‚îú‚îÄ pf2e-adapter.mjs             ‚Üê Pathfinder 2e field mapping
            ‚îî‚îÄ generic-adapter.mjs          ‚Üê Data-driven adapter (any system)
```

### Data Flow

1. **GM starts Foundry** ‚Üí `module.mjs` initializes SyncManager
2. **SyncManager.start()** ‚Üí validates config, initializes modules, connects WS
3. **WebSocket connects** ‚Üí performs initial sync (pulls mappings since `lastSyncTime`)
4. **Live updates** ‚Üí WS messages routed to modules; REST for CRUD operations
5. **Bidirectional** ‚Üí Foundry hooks detect local changes, push to Chronicle via REST

### Sync Guard

All sync modules use a `_syncing` boolean flag to prevent infinite loops:
- Set `true` before applying a remote change to Foundry
- Foundry's change hooks fire but see `_syncing === true` ‚Üí skip pushing back
- Reset to `false` after the change is applied

## WebSocket Message Types Handled

- `entity.created/updated/deleted` ‚Üí JournalSync
- `drawing.created/updated/deleted` ‚Üí MapSync
- `token.created/moved/updated/deleted` ‚Üí MapSync
- `fog.updated` ‚Üí MapSync (renders fog regions as overlay drawings)
- `calendar.event.created/updated/deleted` ‚Üí CalendarSync
- `calendar.date.advanced` ‚Üí CalendarSync
- `sync.status` ‚Üí API client (connection state)

## Security

- API key stored as Foundry world setting (GM-only visible, masked as password input)
- REST uses `Authorization: Bearer <key>` header
- WebSocket uses `?token=<key>` query param (necessary for WS upgrade)
- Per-key rate limiting enforced server-side (60 req/min default)
- Campaign-scoped: key can only access its bound campaign
- Device fingerprint binding: optional one-time device lock

## Known Limitations

- **Fog of war**: One-way sync (Chronicle ‚Üí Foundry). Fog regions rendered as
  semi-transparent polygon drawings on the Foundry scene. No Foundry ‚Üí Chronicle
  push yet (Foundry's native fog is vision-based, Chronicle's is polygon-based)
- **SimpleCalendar events**: Limited ‚Äî events managed as journal notes, no CRUD hooks
- **Shop icon field**: Always returns null (`shop-widget.mjs:146`)
- **Single scene**: Only the active Foundry scene syncs (no multi-scene)
- **GM only**: Full sync runs only for GM users; players get passive updates via Foundry
- **Character sync: limited fields back from PF2e**: Only HP and name sync from Chronicle to PF2e actors (most values are derived from items/rules)
- **Character sync: custom system requirements**: Custom game systems must include `foundry_path` annotations on their character preset fields for bidirectional sync. Systems without annotations get no character sync (the generic adapter returns null). dnd5e and pf2e have hand-written adapters that handle edge cases.
- **System matching fallback**: `SYSTEM_MAP_FALLBACK` (3 entries: dnd5e, pf2e, drawsteel) in sync-manager.mjs is used only when the `/systems` API call fails or the system lacks a `foundry_system_id` in its manifest. Normal operation queries the API and matches dynamically.
